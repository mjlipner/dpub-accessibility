This document, “Digital Publishing Accessibility: How WCAG 2.0 and Other W3C/WAI Guidelines Apply to Digital Publishing” describes how the Web Content Accessibility Guidelines (WCAG) 2.0 [[WCAG20]] and its principles, guidelines, and success criteria can be applied to the needs of Digital Publishing. It provides informative guidance, but does not set requirements.

This document is intended to become a Working Group Note.

Introduction

This document provides informative guidance (but does not set requirements) with regard to interpreting and applying Web Content Accessibility Guidelines (WCAG) 2.0 [WCAG20] to the needs of digital publishing.

While the World Wide Web Consortium (W3C)'s W3C Web Accessibility Initiative (WAI) is primarily concerned with web technologies, its guidance is also relevant to non-web technologies, such as digital publishing published using web-based technological tools.

The current document is focused on the accessibility of digital publishing to people with disabilities and is not intended to supplant any other W3C work.

WCAG 2.0 and Digital Publishing

"Digital Publishing" is a generic term for the broad ecosystem of electronic journals, magazines, news, or book publishing (authors, creators, publishers, news organizations, booksellers, accessibility and internationalization specialists, etc.). The formats used by eBook readers and tablets for electronic books, magazines, journals and educational resources are largely based on W3C Technologies, such as (X)HTML, CSS, SVG, SMIL, MathML, or various Web API-s. Commercial publishers also rely on W3C technologies in their back-end processing all the way from authoring through to delivering the printed or electronic product and beyond. In general one can say that the Publishing Industry is one of the largest communities relying large palette of W3C technologies.

However, the alignment of the needs of the Publishing Industry and the various W3C recommendations is not satisfactory. Necessary features may be missing in the W3C documents, or may be in draft only; as a result, for example, EPUB3, the standard for electronic books, had to introduce its own extensions to cover the needs of publishing.

Other W3C-WAI Guidelines Related to Digital Publishing

UAAG 2.0

The User Agent Accessibility Guidelines (UAAG) 2.0 [UAAG2] is meant for the developers of user agents (e.g. web browsers and e-readers). A user agent that follows UAAG 2.0 will improve accessibility through its own user interface, through options it provides for rendering and interacting with ​content, and through its ability to communicate with other technologies, including assistive technologies.

To assist developers of e-readers, the UAAG 2.0 Reference support document needs to contain examples relevant to digital publishing. It has not been within the scope of this task force to create a note with some examples for the UAAG team. However, it should be a product of future digital publishing accessibility work.

ATAG 2.0

The Authoring Tool Accessibility Guidelines (ATAG) 2.0 [ATAG2] provides guidelines for the developers of authoring tools. An authoring tool that follows ATAG 2.0 will be both more accessible to authors with disabilities (Part A) and designed to enable, support, and promote the production of more accessible web content by all authors (Part B).

To assist developers of authoring tools for digital publishing, the Implementing ATAG 2.0 support document needs to contain examples relevant to digital publishing. It has not been within the scope of this task force to create a note with some examples for the ATAG team. However, it should be a product of future digital publishing accessibility work.

Discussion of Digital Publishing Issues not currently addressed by W3C standards

Page Numbers

Suggestion: UAG suggestion and WCAG suggestion, probably also ATAG suggestion

Comments: For electronic publications -- especially those intended to have direct correspondence to printed "hard-copy" publications such as textbooks -- a way needs to exist for a user to accurately identify the page number, and to navigate accordingly. This method must be accessible, and should not disrupt continuous reading by screen reader users, and should be accurate and accessible for users using Assistive Technologies which may resize or reflow text on the screen.

A method for providing page numbers is addressed in CSS Paged Media Module Level 3 but it applies to the scenario of pagination of text being sent to a printer from a web page. However, there may be something that could be adopted or extended from this model, where rather than having pagination be determined by the dimensions of the paper to be printed upon, a publisher supplies the page dimensions and rules for page numbering in metadata, and the user agent allows the end-user to access the information.

More investigation is necessary.

References

  • CSS Paged Media Module Level 3, 6. Page-Margin Boxes

Drop Cap

Suggestion: UAAG suggestion, WAI outreach suggestion

Comments: The CSS 3 selector first-letter allows the creation of accessible drop caps. However, it is not widely supported. When drop caps are created without using this CSS3 selector, this causes accessibility issues when the two components of the word are read as separate words. Outreach to user agent creators and content creators is needed.

References:

  • The CSS 3 selector first-letter

Notes (Footnotes, Asides, Author Notes, etc.)

Suggestion: No action item

Comments: DPUB-ARIA 1.0 includes dpub-footnote and dpub-footnotes. These roles convey that information is ancillary without conveying information about positioning. This allows authors and user agents to achieve positioning using CSS or scripting while maintaining the flow and navigability of the content.

Phonetic spellings of proper nouns and jargon terms

Suggestion: UAAG and WCAG suggestion

Comments: Publishers need to be able to put in suggested pronunciation for proper nouns, acronyms, and field-specific jargon terms. The Pronunciation Lexicon (PLS) exists but it is exceptionally difficult to use, and doesn't scale well to a large publisher's use cases.

Use cases:

  1. Use Case (WCAG): A chemistry book needs to use the accepted pronunciations of chemistry terms to maximize clarity. It references a pre-existing pronunciation dictionary made available by the American Chemical Society that uses community-agreed-upon standards.
  2. Use Case (WCAG): The trade novel THE LIVES OF CHRISTOPHER CHANT by Diana Wynne Jones, in which there is a plot point that a character's name is spelled "Ralph" but pronounced "Rafe," can be correctly encoded using a local pronunciation dictionary.
  3. UseCase (UAAG): Reading systems support the pronunciation dictionary reference and expose the data to Accessibility APIs.

layout's influence on comprehension

Suggestion: No action item

Comments: There is currently no way to provide visual information about the layout of text on a page that is not semantically determined. Certain literary styles such as poetry may use the placement of text to convey extra meaning. For instance, a poem about a tree may be arranged to look like a tree. There is no good way for someone accessing the text through Assistive Technology which speaks the text, or changes its appearance, to be made aware of this additional information. We have no suggested approaches at this time.

References:

  • Item #2 5/15/2015 DPUB Minutes

Deeply Nested Heading Levels Best Practices

Suggestion: Techniques / Tutorials suggestion

Comments: It is not easy to determine what best practice is for deeply nested headings, headings that go beyond <h6>. Best Practices should be clearly established and easy to find. Use cases in books are more frequent than other circumstances. This is likeliest to occur if an <aside> begins with <h5> or <h6>. It may also occur in encyclopedias and similarly larger and detailed works.

Semantically needed List-Head

Suggestion: WCAG suggestion

Comments: There is no clear method of associating a title of a list with the list items. This can be accomplished using artificial wrappers, such as <div>, but it seems that the list elements (<ul> and <ol>) should include titles. It is tempting to use the h1 – h6 elements, but these have the native semantics of aria role=”heading”, offering unwanted default navigation. It would be useful if there were a role of element called "listtitle" or something similar.

The following is a simple example: 

<ul>
<listtitle>Noble Gases</listtitle>
<li>Helium</li>
<li>Neon</li>
<li>Argon</li>
<li>Krypton</li>
<li>Xenon</li>
<li>Radon</li></ul>

Diagrammar

Diagrammar is a framework to hold multiple description types, in order to make images, tables, graphics accessible. The Diagrammar, defines a data model for image description metadata. It provides a structured, standard way for image description data to be modeled, including multiple different types of descriptions for each image, various descriptions and pointers to tactile graphic files, etc.

For a given image there may be one or more of the following accessibility tags which links to the supplementary information.

  • about
  • summary
  • longdesc
  • simplifiedLanguageDescription
  • tactile
  • braille
  • simplifiedImage
  • tour

Metadata is used to help provide context on what this alternative descriptions and files are. Example of the Metadata available would be

  • Purpose
  • Age / Grade Level
  • Quality rating
  • Content (ie. Links to the descriptions / files etc.)
This metadata could be ultimately searched on to find these alternate image descriptions.

References:

  • Diagram Center Website
  • Diagrammar Schema
